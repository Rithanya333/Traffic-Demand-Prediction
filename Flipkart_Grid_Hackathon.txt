Abstract
   This document provides a comprehensive, elaborated mathematical and techni- cal reference for the Traffic Demand Prediction (TDP) v8 Aggressive model. The model employs a multi-stage pipeline combining sophisticated feature engineering, target encoding, ensemble learning, and post-processing calibration. Traffic demand forecasting is a critical component of intelligent transportation systems, enabling re- source allocation, congestion mitigation, and dynamic routing optimization. This document details all mathematical formulations, loss functions, evaluation met- rics, and implementation considerations across nine base models unified through Optuna-optimized weighting and multi-seed bagging. The "aggressive" designa- tion refers to the extensive feature set, aggressive regularisation tuning, and deep ensemble stacking used to maximize predictive accuracy while controlling overfit- ting. This elaborated version includes additional derivations, numerical examples, expanded theoretical background, and explicit rationales for every design choice, explaining why each component was selected over alternatives.

1 Introduction & Problem Formulation
1.1 Problem Statement
The traffic demand prediction problem is formulated as a supervised regression task. Given a set of contextual features extracted from spatio-temporal, meteorological, and

road network data, the objective is to predict normalized traffic demand specific geographic locations and temporal intervals.

• X ? Rn×p = feature matrix with n samples and p features
• y ? [0, 1]n = normalized target (traffic demand)
• yˆ : Rp ? [0, 1] = prediction function (model)
• L(y, yˆ) = loss function quantifying prediction error
   Selection Rationale for Normalization: Demand counts vary widely across lo- cations and times. Normalizing to [0, 1] ensures that all models operate on a consistent scale, which improves numerical stability and allows fair comparison of errors across dif- ferent geographies. The maximum observed demand is used as the normalizer to preserve the relative magnitude.

1.2 Motivation for Demand Prediction
Accurate traffic demand prediction enables:
1. Resource Optimization: Allocation of vehicles, drivers, and infrastructure based on predicted demand
2. Congestion Mitigation: Proactive routing and capacity planning
3. Revenue Forecasting: Demand-based pricing and surge estimation
4. Operational Efficiency: Reduction of idle time and improved utilization
5. Environmental Impact: Optimized fleet deployment reduces carbon footprint
6. Safety: Anticipating high-demand periods allows pre-positioning of emergency ser- vices
   Selection Rationale for These Use Cases: These five areas represent the pri- mary stakeholders' needs identified through business requirements analysis. Resource optimization directly impacts operational costs; congestion mitigation is a regulatory pri- ority; revenue forecasting drives financial planning; operational efficiency affects driver satisfaction; environmental impact aligns with sustainability goals. Safety was added as an emerging benefit.

2.2 Timestamp Parsing & Temporal Decomposition
Timestamps are parsed from ISO 8601 format and decomposed into constituent temporal components. Each component captures different periodicities in demand patterns.

2.2.1 Hour Extraction
h = 1 timestamp in seconds I mod 24,	h ? {0, 1, . . . , 23}	(5)
   Purpose: Extract hourly component; captures morning peak (07:00-09:00), after- noon (12:00-17:00), evening peak (17:00-19:00), and off-peak periods.
   Rationale: Demand exhibits strong hourly periodicity correlated with commute pat- terns, meal times, and business hours.
   Selection Rationale: Hour is extracted as an integer from 0-23. This discrete representation allows tree models to split on specific hours (e.g., hour=8 vs hour=20) without imposing linearity, unlike a continuous time feature which would force monotonic relationships.

2.2.2 Minute Extraction
m = timestamp in seconds mod 3600 ÷ 60,	m ? {0, 1, . . . , 59}	(6)
Purpose: Extract minute component for fine-grained temporal resolution.

   Rationale: Within-hour demand variations can be significant in high-frequency set- tings; enables 15-minute slot discretization.
   Selection Rationale: Minute precision is retained because demand can change rapidly (e.g., after a concert ends). However, using minutes directly would create 1440 categories, which is too sparse. Thus we aggregate to 15-minute slots.

2.2.3 Continuous Time-of-Day Feature
tmin = 60h + m ? [0, 1440),	tmin ? R	(7)
   Purpose: Convert time into a continuous scale (minutes since midnight) for harmonic (Fourier) feature generation.

   Selection Rationale: Continuous time is essential for trigonometric transforma- tions. Using minutes instead of seconds reduces magnitude while preserving precision for harmonic features. The range [0, 1440) naturally wraps around midnight.

2.3 Day-Level Temporal Features
2.3.1 Day-of-Week Extraction
dow = (date - epoch) mod 7,	dow ? {0, 1, 2, 3, 4, 5, 6}	(11)
Convention: 0 = Monday, 1 = Tuesday, ..., 6 = Sunday.
   Purpose: Capture weekly demand cycles (weekday vs. weekend effects). Many transportation systems exhibit a 7-day periodicity, with reduced demand on weekends and increased demand on weekdays.
   Selection Rationale: Day-of-week as a categorical variable (0-6) allows tree models to treat each day separately, capturing fine-grained differences (e.g., Monday rush vs Friday rush). A binary weekend flag alone would lose this nuance. Monday-Sunday ordering is arbitrary; the model will learn splits.

2.3.2 Day-of-Year Extraction
doy = (date - year start) ? [1, 365 or 366]	(12)
   Purpose: Capture long-term seasonal and holiday effects. For example, demand near Christmas or summer holidays differs from regular days.
   Selection Rationale: Day-of-year is used as input to sine/cosine harmonics rather than directly because raw DOY (1-365) would impose a discontinuity between Dec 31 and Jan 1. Using harmonics avoids this boundary issue and models smooth seasonal transitions.
Leap year handling: The day-of-year range is 1-366, and the harmonic features use
365.25 as the period to smooth over leap years.

9.2 Stratification Considerations
For imbalanced regression (rare high/low demand periods), stratified splitting ensures balanced fold composition.

9.2.1 Demand Quantile Stratification
                      stratai = quantile(yi, q = 5)	(89) Divide targets into 5 quantile-based bins and ensure each fold has proportional rep-
resentation.
   Benefit: Each fold contains similar demand distributions, improving cross-fold com- parability.

   Selection Rationale: Without stratification, one fold could contain all high-demand samples (e.g., a holiday week), leading to inflated variance in fold scores. 5 quantile bins is a natural choice for 5-fold CV.

11 Data Leakage Prevention
Data leakage-inadvertent use of future or validation data during training-is a critical concern in time-series modeling. Multiple safeguards are implemented.

11.1 Fold-Safe Target Encoding Protocol
11.1.1 Leakage Prevention During Target Encoding
TEfold f = compute from(Trainf ) ? apply to(Valf , Test)	(92)
Procedure for Fold f :
1. Fit target encoding statistics (mean, std) on Trainf only
2. Apply fitted encoding to both Valf and Test using statistics from step 1
3. Never fit encoding on Valf or Test
Mathematical Guarantee:
         P(leakage during TE) = 0  (no validation data in fit step)	(93)
   Selection Rationale: Without this, target encoding would use the target values of validation samples to compute the encoding for those same samples, leading to severe overfitting (R² could be inflated by 0.1-0.3). The fold-safe protocol is non-negotiable for production models.

11.2 Day-48 Lag Masking
The d48 demand lag feature references demand from exactly 48 days prior. On day 48 of the series, this would reference non-existent future data.

11.2.1 Prevent Data Leakage for Lag Feature

:=	NaN	if dayi = 48 value from day 48 prior	 otherwise

   Rationale: On day 48, if we used the actual demand, we'd use data from the future (day 96). NaN masking prevents this.
   Impact: Tree-based models handle NaN by routing samples to default directions; minimal performance impact.
   Selection Rationale: Setting to NaN is the safest approach; models can learn a default branch for this case. An alternative would be to drop day 48 samples, but that would lose data. The lag was shifted from 49 to 48 days to avoid this issue in the first place? Actually, we keep it but mask.

11.3 Hyperparameter Validation
11.3.1 No Hyperparameter Tuning on Test Data
All hyperparameters are tuned on training + validation data only. Test data is touched only for final evaluation.

Hyperparameters are optimized over {Train, Val} only	(95)

11.3.2 Cross-Validation Boundaries
Each fold's hyperparameters are fit on its training portion; applied uniformly to validation and test.
   No Peeking: Models never see metrics or data from later time periods during train- ing.
   Selection Rationale: This is a fundamental principle of machine learning. Violating it leads to overly optimistic performance estimates that do not generalize to production.

D Debugging Checklist
1. High RMSE: Check for missing data, outliers, or class imbalance
2. Overfitting: If train R²	test R², increase regularisation or reduce model capac- ity
3. Underfitting: If train R²	test R²	0, increase model complexity or feature engineering
4. Data Leakage: Verify target encoding is fold-safe; check lag features for future references
5. NaN Predictions: Investigate extreme feature values; apply clipping to outputs
6. Slow Inference: Profile feature computation; consider feature selection or dimen- sionality reduction
